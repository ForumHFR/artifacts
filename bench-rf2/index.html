Bench Gradle — Redface 2

Mesurez le temps de build/tests de Redface 2 sur votre machine et contribuez à une table comparative CPU/RAM/disque pour repérer les zones lentes du build et le potentiel d'accélération.

Prérequis

À lire avant de lancer le script
  • git, un JDK 17 ou plus récent, un SDK Android déjà installé (ANDROID_HOME / ANDROID_SDK_ROOT ou emplacement par défaut — le script ne l'installe pas).
  • ~10 Go d'espace disque libre pour le premier run (cache Gradle, artefacts de build).
  • Premier run = plusieurs Go de téléchargements réseau (distribution Gradle, dépendances du projet). Durée : grossièrement 30 minutes à froid, quelques minutes en --full déjà chaud.
  • Ne benchez pas sur batterie : branchez la machine, évitez toute autre charge CPU en parallèle. Les résultats varient d'un run à l'autre — une mesure isolée est indicative, pas une vérité absolue.

Comment lancer

  1. Téléchargez le script, hébergé à côté de cette page : 
    curl -O https://forumhfr.github.io/artifacts/bench-rf2/bench-rf2.sh
    (ou clic droit → enregistrer le lien).
  2. Lisez le script avant de l'exécuter (bonne pratique générale, et il est commenté en tête de fichier). Il clone une copie dédiée sous ~/.cache/bench-rf2, sans toucher à votre propre checkout, et ne pousse ni ne publie rien lui-même.
  3. Rendez-le exécutable et lancez-le : 
    chmod +x bench-rf2.sh
./bench-rf2.sh          # scénarios S1 à S4
./bench-rf2.sh --full   # ajoute S5a/S5b (detekt + lint, comme la CI)
  4. À la fin, le script affiche un bloc de résultats en Markdown et en BBCode HFR (infos machine + temps par scénario), aussi écrit sous ~/.cache/bench-rf2/results/<horodatage>/.
  5. Postez le bloc BBCode (prêt pour un forum, encadré en [fixed]) sur le topic Redface 2 sur ForumHFR. Copier-coller manuel : le script n'envoie rien lui-même.

Scénarios mesurés

Chaque scénario est précédé d'un ./gradlew --stop (démon frais), pour des temps comparables entre machines.

ScénarioCommandeRévèle
S1 — configuration + build-logic ./gradlew help (cache de configuration purgé) Temps de configuration Gradle et de l'included build build-logic (peut être recompilé ici) — pas de compilation applicative. La distribution Gradle est déjà téléchargée avant ce scénario (warm-up), hors chronométrage.
S2 — clean build ./gradlew clean :app:assembleProdDebug --no-build-cache Coût complet depuis zéro : configuration + KSP/Hilt + compilation + packaging.
S3 — warm no-op même commande que S2, sans clean Coût des vérifications d'incrémentalité (up-to-date checks) quand rien n'a changé.
S4 — tests unitaires ./gradlew test testDebugUnitTest Tous les tests JVM unitaires du dépôt — les tâches réelles de la CI.
S5a — detekt --full ./gradlew detektAll Coût de l'analyse statique detekt (tâche générique qui délègue à la vraie JavaExec detektCliCheck, une seule passe sur tout le dépôt — pas de décomposition par module).
S5b — lint --full ./gradlew lintDebug :app:lintProdDebug Coût du lint Android (:app est flavoré « channel », lintDebug seul n'y résout pas de variante).

Pistes d'accélération

Actif aujourd'hui : parallélisme local (org.gradle.parallel), cache de build local (org.gradle.caching — aucun cache distant/partagé configuré), cache de configuration, heap démon à 4 Go, et fan-out CI en 4 jobs parallèles (assemble/test/lint/detekt).

À l'étude :

  • Cache de build distant/partagé entre machines et CI (impact surtout sur S2/S4).
  • Exécution de tests déportée/répartie par module plutôt qu'en un seul appel.
  • Fan-out CI plus fin que les 4 jobs actuels.
  • Profilage KSP/Hilt dédié (--scan ou équivalent), hors périmètre de ce script volontairement simple.
