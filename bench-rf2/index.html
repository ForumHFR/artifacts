Bench Gradle — Redface 2

Mesurez le temps de build/tests de Redface 2 sur votre machine (CPU, RAM, disque) pour aider à identifier les zones lentes du build et le potentiel d'accélération (parallélisme local, cache de build, workload déporté).

Redface 2 est un client Android open source pour un forum, construit avec Kotlin/Compose/Hilt/Gradle multi-module. Ce petit protocole de bench a un but précis : constituer une table comparative « machine → temps de build » à partir de contributions volontaires, pour répondre à trois questions concrètes :

  1. Où le temps de build part-il réellement (configuration Gradle, KSP/Hilt, compilation Kotlin, tests unitaires, lint/detekt) ?
  2. Quel est l'effet du cache de build et du parallélisme local sur des machines variées ?
  3. Y a-t-il un potentiel réaliste à déporter une partie du travail (cache de build partagé, exécution de tests à distance, CI plus parallèle) ?

Prérequis et avertissements

À lire avant de lancer le script
  • git, un JDK 17 ou plus récent, et un SDK Android déjà installé (le script ne l'installe pas — il vérifie ANDROID_HOME/ANDROID_SDK_ROOT ou quelques emplacements par défaut).
  • ~10 Go d'espace disque libre recommandés pour le premier run (paquets SDK éventuellement manquants, cache de dépendances Gradle, artefacts de build). Les runs suivants sont beaucoup plus légers en réseau/disque.
  • Premier run = téléchargements réseau de plusieurs Go : distribution Gradle (~200 Mo), dépendances du projet (plusieurs centaines de Mo), et éventuellement des paquets SDK Android si votre installation ne les a pas déjà. Comptez une bonne connexion et de la patience.
  • Durée totale estimée : très variable selon la machine et l'état du cache local — grossièrement de 30 minutes (petite machine, premier run, réseau moyen) à quelques minutes seulement pour un run --full déjà « chaud ». Le script chronomètre chaque étape séparément, ce n'est pas une boîte noire de durée fixe.
  • Ne benchez pas sur batterie : le throttling thermique/énergie d'un laptop sur batterie fausse complètement la mesure. Branchez la machine, évitez de lancer d'autres charges CPU en parallèle, et si possible désactivez temporairement les gouverneurs d'économie d'énergie agressifs.
  • Les résultats varient d'un run à l'autre (throttling thermique, autres process, état du cache). Une seule mesure est indicative, pas une vérité absolue — voir les limites en fin de page.

Instructions pas-à-pas

  1. Téléchargez le script, hébergé à côté de cette page : 
    curl -O https://forumhfr.github.io/artifacts/bench-rf2/bench-rf2.sh
    (ou clic droit → enregistrer le lien depuis votre navigateur).
  2. Lisez le script avant de l'exécuter. C'est une bonne pratique générale avant de lancer n'importe quel script téléchargé sur votre machine — celui-ci est commenté en tête de fichier pour faciliter la relecture. Il ne fait que : cloner une copie dédiée du dépôt sous ~/.cache/bench-rf2, lancer des builds Gradle chronométrés, et afficher un résultat en local. Il ne pousse et ne publie rien lui-même.
  3. Rendez-le exécutable et lancez-le : 
    chmod +x bench-rf2.sh
./bench-rf2.sh          # scénarios S1 à S4
./bench-rf2.sh --full   # ajoute S5 (detekt + lint, comme la CI)
  4. À la fin, le script affiche un bloc de résultats en deux formats (Markdown et BBCode HFR), avec toutes les infos machine et les temps par scénario. Les fichiers sont aussi écrits sous ~/.cache/bench-rf2/results/<horodatage>/.
  5. Postez le bloc BBCode (déjà prêt pour un forum HFR, encadré en bloc via [fixed]) sur le topic Redface 2 sur ForumHFR. C'est un copier-coller manuel : le script n'envoie rien lui-même.

Que mesure chaque scénario ?

Chaque scénario est précédé d'un ./gradlew --stop (démon Gradle frais) et d'un état de cache contrôlé, pour que les temps soient comparables entre machines.

S1 — cold configure

./gradlew help après purge du cache de configuration local au projet.

Révèle : le temps de configuration pure du graphe de tâches (évaluation des build.gradle.kts, résolution des plugins de convention build-logic, aucune compilation). Une configuration lente pointe vers un projet Gradle trop chargé en logique de configuration plutôt qu'un problème de compilation.

S2 — clean build

./gradlew clean :app:assembleProdDebug --no-build-cache, démon frais, cache de build désactivé.

Révèle : le coût complet d'un build depuis zéro — configuration + KSP/Hilt (génération de code par annotation) + compilation Kotlin de tous les modules dont dépend :app + packaging APK. C'est le pire cas réaliste (nouvelle machine, cache vidé, CI sans cache chaud).

S3 — warm no-op

Même commande que S2, sans clean : ./gradlew :app:assembleProdDebug --no-build-cache, exécutée immédiatement après S2.

Révèle : le coût des vérifications d'incrémentalité de Gradle (up-to-date checks) quand rien n'a changé. Un temps élevé ici indique que le graphe de tâches est coûteux à re-vérifier même sans travail réel à faire — un signal utile indépendamment de la vitesse de compilation brute.

S4 — tests unitaires

./gradlew test testDebugUnitTest — exactement les tâches utilisées par la CI du projet (voir .github/workflows/ci.yml) : test couvre les modules JVM purs (core:model, core:domain, core:parser, …) et agrège les variantes de :app ; testDebugUnitTest couvre les modules Android non flavorés (core:database, core:ui, core:network, core:data, core:extension, chaque feature:*).

Révèle : le coût de tous les tests unitaires du dépôt, pas seulement ceux de :app — c'est le scénario qui ressemble le plus à ce qu'un contributeur attend avant de pousser une PR.

S5 — detekt + lint optionnel, --full

./gradlew detektAll puis ./gradlew lintDebug :app:lintProdDebug — les deux commandes utilisées par la CI. detektAll est une tâche custom qui lance detekt-cli en une seule passe sur tout le dépôt (pas une agrégation par module) ; lintDebug couvre les modules non flavorés et :app:lintProdDebug est nécessaire en plus car :app a une dimension de flavor (channel : prod/beta/dev) — le lintDebug non qualifié ne résout pas de variante pour ce module.

Révèle : le coût de l'analyse statique, souvent négligé dans les discussions de perf de build mais bien présent dans la CI réelle du projet (deux des quatre jobs de la matrice CI).

Pistes d'accélération étudiées

État établi en lisant gradle.properties, .github/workflows/ci.yml et settings.gradle.kts du dépôt — pas des suppositions.

Actif aujourd'hui actif

  • org.gradle.parallel=true — exécution parallèle des tâches indépendantes entre modules, en local.
  • org.gradle.caching=true — cache de build local activé (réutilisation des sorties de tâches déjà exécutées sur la même machine). Aucun cache de build distant/partagé n'est configuré dans settings.gradle.kts à ce jour.
  • org.gradle.configuration-cache=true — cache de configuration activé (évite de re-évaluer les scripts Gradle à chaque invocation quand rien n'a changé en amont).
  • org.gradle.jvmargs=-Xmx4g — heap du démon Gradle fixé à 4 Go.
  • Fan-out CI par type de tâche : la CI (ci.yml) ne lance plus un seul job séquentiel mais 4 jobs en parallèle (assemble / test / lint / detekt), chacun avec son propre cache incrémental via gradle/actions/setup-gradle — un moyen déjà en place de dépasser le plafond de vCPU d'un runner unique.

Pistes non actives, à l'étude piste

  • Cache de build distant/partagé (nœud Develocity Build Cache, ou cache HTTP auto-hébergé) : permettrait aux contributeurs et à la CI de partager les sorties de tâches entre machines, au lieu d'un cache local par poste. Impact potentiel le plus net sur S2/S4 pour les modules inchangés d'une PR à l'autre.
  • Exécution de tests déportée (test distribution / matrice de tests plus fine) : aujourd'hui S4 lance test testDebugUnitTest en un seul appel couvrant tous les modules dans le même job CI ; répartir les tests par module ou groupe de modules sur plusieurs jobs/machines réduirait le temps mur.
  • Fan-out CI plus fin : la matrice actuelle a 4 entrées ; un découpage plus fin (par exemple par groupe de modules pour les tests) est une piste pour aller plus loin que le fan-out déjà en place.
  • KSP/Hilt : la génération de code par annotation (Hilt via KSP2) est un point chaud classique des builds Android/Kotlin. Ce bench (en particulier l'écart S2 vs S3) donne une première mesure exploitable ; creuser plus finement demanderait un profil de build dédié (--scan ou équivalent), hors du périmètre de ce script volontairement simple.
